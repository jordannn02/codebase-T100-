# T100 工作流

這份文件描述 `codebase-T100版` 的預設調查方式。它適合用在 T100 / TIPTOP / Genero 4GL 專案，但不把 graph 結果當成唯一證據。

## 一、輸入錨點

開始前先整理問題錨點：

- 程式 ID：例如 `axmt500`、`cxmta41`、`asft311`。
- 表名與欄位：例如 `xmda_t`、`\[客戶編號](xmda004)`。
- 文件或單據：只在私人工作區使用；公開範例請改成假資料。
- 環境邊界：正式、測試、site、ENT，但公開版不要放真實連線資訊。
- 目前證據等級：需求描述、畫面、SQL、source、手冊、會議紀錄。

## 二、圖譜第一輪

先用 code graph 找方向：

- 搜尋程式 ID 或函式名。
- 找同檔案附近函式與相鄰呼叫。
- 找 READS / WRITES / CALLS 類關係。
- 找可能的上游 caller 與下游 callee。
- 找表欄位在 source 中的使用點。

這一步的結果只代表「可能相關」，還不是結論。

## 三、原始碼確認

拿圖譜結果回到原始碼片段確認：

- 條件式是否真的會進入該分支。
- 是否有 site / ENT / 單別 / 狀態碼過濾。
- 是否有標準與客製兩套邏輯。
- 是否有共用函式或 `csub_*` 改寫。
- 欄位是來源、暫存、計算結果，還是畫面顯示用。

## 四、證據分層輸出

回答時建議拆成：

- 結論：目前可確認的行為。
- Source 證據：檔案、函式、條件、呼叫路徑。
- Schema / DB 證據：表名、欄位、只讀 SQL 口徑。
- Runtime / 畫面證據：若有驗證才寫。
- 推論與待確認：不要混在已證實結論裡。

## 五、T100 欄位格式

欄位請用可讀格式：

```text
\[欄位名稱](field_code)
```

例如：

```text
\[客戶編號](xmda004)
\[交易含稅金額](xmab009)
\[據點匯率](xmab013)
```

如果欄位名稱尚未確認，請保留不確定性：

```text
\[欄位名稱待確認](xmab999)
疑似 \[訂單狀態](xmda999)
```

## 六、公開安全規則

公開文件、issue、README、範例 prompt 中不要放：

- 真實客戶 source。
- 正式 DB 查詢結果。
- 內部 IP、帳號、Keychain service、VPN route。
- 單據號、客戶名、供應商名、品號。
- 截圖、transcript、會議紀錄原文。

需要展示時，請改成合成資料或假代號。
